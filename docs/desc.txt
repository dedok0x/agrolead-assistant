Ниже полный «контекст-паспорт» текущего проекта в том виде, как он сейчас устроен в репозитории, чтобы другой ИИ мог сразу работать без догадок.

1) Что это за проект

Проект: локальный B2B sales-assistant для зернового трейдера.
Стек: FastAPI + PostgreSQL + Nginx + Docker Compose + Ollama + NanoClaw adapter.
Ключевой принцип текущей версии: только локальная LLM через Ollama, без внешних API.
Основной пользовательский сценарий: клиент пишет в чат → бот квалифицирует лид (товар/класс/объем/регион/срок/контакт) → передача менеджеру.
Есть админка для управления номенклатурой, промптами, сценариями, лидами, данными компании.

6) Backend: модульная декомпозиция

backend/app/main.py
Точка входа FastAPI.
Инициализация:
llm_service = LLMService()
nanoclaw = NanoClawAdapter()
На startup:
init_db() (create_all)
seed_defaults() (дефолтные данные)
Главная оркестрация диалога в process_chat():
создает/находит сессию
пишет user-message
запускает guardrails
обновляет state/lead (sales_logic)
если не хватает полей — возвращает следующий вопрос
если квалифицировано — идет через nanoclaw-agent
если agent упал — fallback напрямую в llm_service
записывает assistant-message + состояние.
Есть совместимость endpoint /api/picoclaw/agent/chat как deprecated-обертка.
backend/app/db.py
DATABASE_URL берется из env.
Если SQLite — check_same_thread=False.
init_db() через SQLModel.metadata.create_all(engine).
Миграций Alembic нет, только create_all.
backend/app/models.py (SQLModel, table=True)
Таблицы:
CompanyProfile
PromptCategory
Scenario
ScenarioTemplate
ChatSession
ChatMessage
ConversationState
ProductItem
Lead
Примечание: явных FK-констрейнтов нет, связи логические по session_id.
backend/app/guardrails.py
Regex-based safety.
Блоки:
severe toxic (иди на х..., etc.) → hard stop
mild toxic (бля, сука, ...) → soft stop в strict режиме
security (ddos, botnet, xss, sql injection, взлом) → hard security block
Возвращает GuardrailDecision.
backend/app/sales_logic.py
Извлечение из текста:
product, grade, volume, region, delivery_term, contact
State-machine:
greeting -> qualification -> offer -> handoff
отдельный stopped_toxic
sync_state_and_lead():
обновляет ConversationState и Lead
выставляет missing_fields
синхронизирует ChatSession.last_state.
next_qualification_question() задает строго следующий вопрос.
backend/app/llm_service.py
Сейчас только Ollama.
Стратегия endpoint compatibility:
пробует по очереди:
/api/generate
/api/chat
/v1/chat/completions
Ведет метрики last provider/model/error/usage.
status() возвращает preferred_provider=ollama.
На ошибке бросает LLMUnavailableError.
backend/app/nanoclaw_adapter.py
API-клиент к nanoclaw-agent.
Endpoint по env: NANOCLAW_BASE_URL + NANOCLAW_AGENT_CHAT_PATH.
Передает message + context.
Валидирует непустой текст ответа.
backend/app/seed.py
Идемпотентный seed:
профиль компании
prompt categories (identity/scope/safety/style/lead_capture)
сценарии
номенклатура зерна
шаблоны сценариев.
Есть reset_default_scenario_templates().
7) Схема данных (БД) — практический смысл

ChatSession
1 запись на диалоговую сессию клиента.
ключи: id, client_id, last_state, timestamps.
ChatMessage
история сообщений.
поля диагностики: blocked, reason, provider, model_used.
ConversationState
«оперативная память» state-machine на сессию.
уникальный session_id.
собранные поля лида + missing_fields.
Lead
бизнес-сущность лида для менеджера.
статус: new/in_progress/qualified/blocked... (зависит от логики и админки).
ProductItem
витрина товаров/цен/остатков.
PromptCategory
редактируемые системные промпты из админки.
CompanyProfile, Scenario, ScenarioTemplate
публичная инфа и контент для UI/сценариев.

11) Deploy pipeline (deploy.sh)

Ключевая логика:

Проверяет инструменты: git, docker, curl, python.
Делает backup .env в /tmp.
Жестко чистит старое:
docker compose down -v --rmi all --remove-orphans
удаление legacy container/image/volume/dirs.
Синхронизирует git с текущей веткой:
fetch
reset --hard origin/<current-branch>
clean -fdx
Восстанавливает .env из backup, либо создает из example.
Принудительно фиксирует ollama-only конфиг в .env:
LLM_PROVIDER=ollama
NANOCLAW_DEFAULT_PROVIDER=ollama
OLLAMA_MODEL=tinyllama
OLLAMA_NUM_CTX=512
OLLAMA_NUM_PREDICT=96
docker compose build --no-cache --pull
docker compose up -d --force-recreate
Health checks:
/api/health
PostgreSQL pg_isready
Ollama /api/tags
ollama pull модели
/api/llm/status
/api/chat/dry-run
/api/nanoclaw/agent/chat
smoke диалоги
автотесты backend/tests/*
Если dry-run падает с memory error — авто-снижение до OLLAMA_NUM_CTX=256, OLLAMA_NUM_PREDICT=64 и рестарт API+agent.
На любой ошибке печатает последние логи контейнеров и прерывает деплой.
В конце выводит ✅ DEPLOY SUCCESS.
Wrapper:

deploy/deploy.sh просто делает exec bash ../deploy.sh.

14) Важные ограничения/риски (для следующего ИИ)

Нет миграций (Alembic): схема создается create_all.
Нет FK/relational constraints между таблицами.
Админ auth простой:
логин/пароль из env,
статический token без expiry,
хранится в localStorage на фронте.
Inference синхронный и inline (нет очередей).
Нет централизованных метрик/трейсов/алертов.
На low-memory серверах критичны OLLAMA_NUM_CTX и выбранная модель.
app.version в main.py всё еще 3.0.0, хотя docs уже v4.
prompt_secret_if_empty() в deploy.sh остался как helper, но в текущем пути ollama-only не используется.
15) Минимальный контекст-блок для передачи другому ИИ (копипаста)

Project: agrolead-assistant, local-only LLM architecture.
Stack: FastAPI + PostgreSQL + Nginx + Ollama + NanoClaw HTTP adapter, Docker Compose.
No external LLM providers; provider mode forced to ollama.
Default model: tinyllama, low-memory settings (num_ctx=512, num_predict=96), auto-degrade to 256/64 in deploy if memory error.
Main flow: /api/chat -> guardrails -> state-machine lead capture -> if qualified call nanoclaw-agent -> adapter endpoint /api/nanoclaw/agent/chat -> Ollama.
DB via SQLModel create_all at startup; seeded defaults in seed.py.
Admin endpoints protected by x-admin-token from env; token issued by /api/admin/login.
