# 07. Глубокий разбор легаси и план улучшений

Этот документ фиксирует текущие технические долги и приоритизированный план стабилизации.

## 1) Архитектурные проблемы

### 1.1 Сверхконцентрация логики в `main.py`

- Симптом: один файл содержит маршруты, orchestration, CRUD, часть доменной логики.
- Риск: высокая цена изменений, конфликтные PR, сложно покрывать тестами точечно.
- Что делать:
  1. вынести chat use-case слой (`application/chat_service.py`),
  2. вынести admin use-case слой,
  3. оставить в `main.py` только transport + dependency wiring.

### 1.2 Слабые модульные границы

- Симптом: доменные операции напрямую смешаны с SQLSession.
- Риск: невозможно быстро заменить хранилище или выделить сервисы.
- Что делать: ввести интерфейсы репозиториев и DTO на границе слоев.

## 2) Проблемы в логике диалога и промптов

### 2.1 Настройки intake/routing не управляют backend-логикой

- Симптом: `admin_setting` хранит `intake.sequence.*` и `routing.rule.*`, но backend почти не использует.
- Риск: ложное ожидание у бизнеса, что настройки реально влияют на оркестрацию.
- Что делать: либо подключить эти настройки в runtime, либо убрать из UI до реализации.

### 2.2 Эвристический extraction без fallback-классификатора

- Симптом: регэкспы и keyword-матчинг дают шум на коротких фразах.
- Риск: неверный request type, неправильные missing fields и статус лида.
- Что делать:
  1. добавить confidence gating на тип запроса,
  2. добавить lightweight classifier (rule+ML/LLM consensus),
  3. вести метрики качества extraction.

### 2.3 Частичная рассинхронизация state-моделей

- Есть `lead.status_code`, `chat.current_state_code`, `negotiation_stage`.
- Риск: сложность поддержки, трудно объяснить единый state machine.
- Что делать: формализовать один canonical state chart + mapping table.

## 3) Проблемы модели данных

### 3.1 Отсутствуют миграции

- Симптом: только `create_all`.
- Риск: невозможность безопасно эволюционировать продовую схему.
- Что делать: ввести Alembic, baseline migration, policy schema-change.

### 3.2 Неполная референциальная целостность

- Симптом: много id-полей без строгих FK.
- Риск: orphan-данные, ошибки аналитики.
- Что делать: поэтапно добавить FK и индексы на ключевые связи.

### 3.3 Нехватка историчности

- Симптом: lead хранит в основном текущее состояние.
- Риск: сложно делать качественную воронку и RCA по потерям.
- Что делать: добавить таблицу истории статусов/изменений и событийный журнал.

## 4) API и производительность

### 4.1 Широкий `dict[str, Any]` в CRUD

- Риск: слабая валидация, неочевидные контракты.
- Что делать: strict pydantic schemas + contract tests.

### 4.2 Full-table scan в application-логике

- Симптом: `select(...).all()` + фильтрация в Python для stats/leads/sessions.
- Риск: деградация при росте данных.
- Что делать: фильтрация/агрегации в SQL, пагинация и лимиты.

### 4.3 Несогласованность error semantics

- Пример: dry-run иногда 503, обычный чат может дать 200 с fallback.
- Что делать: единая матрица ошибок + единый error envelope.

## 5) Безопасность

### 5.1 Риски auth-поверхности

- Есть fallback креды по умолчанию и опциональный static token mode.
- Нет rate limit на login.
- Что делать:
  1. mandatory secret policy в production,
  2. login rate limit + lockout,
  3. аудит попыток входа.

### 5.2 Отсутствие role-based authorization

- Сейчас токен валидируется, но права ролей почти не разделяются.
- Что делать: RBAC policy matrix по endpoint'ам.

## 6) DevOps и эксплуатация

### 6.1 Тяжелый деплой

- `deploy.sh` всегда делает `build --no-cache --pull`.
- Риск: долгие релизы и избыточная нагрузка на CI/registry.
- Что делать: staged build, тегирование образов, кешируемая сборка.

### 6.2 Наблюдаемость

- Нет централизованных метрик/tracing.
- Что делать: structured logging + метрики latency/error rate + базовые alert rules.

### 6.3 Отсутствие DR-практик

- Нет документированного backup/restore для PostgreSQL.
- Что делать: регулярный dump, проверка восстановления, retention policy.

## 7) Приоритизированный план

## P0 (1-2 недели): стабилизация

1. Зафиксировать production policy по админ-авторизации (без static token).
2. Ввести строгие pydantic-модели на критичных endpoint'ах (`chat`, `login`, `lead update`).
3. Привести LLM-error поведение к единому контракту.
4. Добавить SQL-пагинацию для `leads` и `chat-sessions`.
5. Убрать/пометить в UI настройки, не влияющие на runtime.

## P1 (2-6 недель): технический фундамент

1. Запустить Alembic и baseline migration.
2. Рефакторинг `main.py` на слои (router/service/repository).
3. Ввести ключевые FK/индексы.
4. Добавить audit trail по изменениям лида.
5. Добавить rate limiting на admin login.

## P2 (6-12 недель): масштабирование

1. Улучшить extraction pipeline (confidence + fallback classifier).
2. Перевести тяжелые агрегации в SQL/materialized views.
3. Ввести telemetry stack (metrics + traces + dashboards).
4. Подготовить сервисные границы для возможного split.

## 8) Целевой образ архитектуры (эволюционно)

1. `api` слой: только HTTP/DTO.
2. `application` слой: сценарии (`ProcessChat`, `UpdateLead`, ...).
3. `domain` слой: state machine, qualification policy, guardrails contract.
4. `infra` слой: SQL repos, LLM adapters, retrieval.

Важно: внедрять поэтапно, не ломая текущий MVP-поток в проде.
